## 1. Main Behavior Rules

### 1.1 System Prompt Overview

The system prompt is loaded once at startup from `src/prompts/systemPrompt.txt` and injected into every Claude API call as the `system` parameter.  
It defines:

- **Identity**: The bot is נטלי, a human sales coordinator. Never admits to being a bot.
- **Tone**: Warm, professional Israeli WhatsApp style. Short messages. 1–2 emojis.
- **Language**: Hebrew by default. Switches to Russian or English if the customer writes in those languages.
- **Format**: Always outputs raw JSON (no markdown, no code fences). The JSON is parsed by the bot.

### 1.2 Greeting Rules

- `"היי, תודה שפניתם לדלתות מיכאל"` appears **only once** — in the very first bot reply.
- The greeting includes the current Israeli time-of-day salutation (בוקר טוב / צהריים טובים / ערב טוב / לילה טוב), computed at call time using `Asia/Jerusalem` timezone.
- The **company pitch** is injected immediately after the greeting on the first reply, before addressing the customer's question:
  > "אנחנו מציעים דלתות כניסה ופנים באיכות הגבוהה ביותר בשוק — מגוון רחב של דגמים ועיצובים בהתאמה אישית, אחריות מקיפה של מעל שנתיים, ואולם תצוגה מרשים בנתיבות. 🚪✨"
- **After the first reply:** no greeting phrase, no pitch. Continue naturally.
- Bot does **not** introduce itself by name unless directly asked ("מי את?").

### 1.3 Main Conversation Rules

| Rule | Description |
|------|-------------|
| ONE question per message | Never stack two questions. |
| Never repeat given info | Never ask for something the customer already provided. |
| Never quote prices | Price is always described as "מותאם אישית לפי סוג ועיצוב". |
| Never quote delivery times | Not in the prompt or FAQ. |
| Gender consistency | Detect gender from verbs/adjectives/name on first opportunity, then lock it for the entire conversation. Unknown → neutral plural. |
| Spelling errors | Always infer from context. Never say "לא הבנתי". |
| Short customer replies | "כן", "לא", "מודרנית" — accept immediately, move to next step. |
| Prohibited bot phrases | "אנא ספק", "המערכת תעבד", "כנציגת השירות", "פנייתך חשובה לנו", "אין לי מידע" — forbidden. |
| Style variety | Never start two consecutive replies the same way. Vary openers and phrasing. |

### 1.4 Follow-up Message Rules

After the bot sends a reply, a timer starts. If the customer goes silent:

- **After 15 minutes:** A personalized follow-up is sent. It references the conversation topic (generated by `claude-haiku-4-5-20251001` using the last 6 turns of history). Fallback if Claude fails: generic "רק רצינו לוודא שקיבלת את כל המידע 😊"
- **After 7 more minutes (22 min total):** Auto-close message is sent:
  > "מכיוון שלא קיבלנו תגובה, נסגור את הפנייה לעת עתה 🙂\nאם תרצו לחזור אלינו בכל עת — אנחנו כאן!\nדלתות מיכאל | 054-2787578"
- Conversation marked `closed: true`. Summary generated and saved to leads file.
- **Follow-up is skipped** if the last bot message was an error reply (prevents piling on after failures).
- Follow-up **is not sent** to group chats (`@g.us` sender IDs).

### 1.5 Fallback Rules

| Trigger | Bot sends |
|---------|-----------|
| Claude API error (timeout, 5xx, rate limit) | "מצטערים, אירעה תקלה זמנית. אנא נסו שנית בעוד רגע." |
| Claude returns invalid / unparseable JSON | "מצטערים, אירעה תקלה זמנית. אנא נסו שנית בעוד רגע 🙏" |
| Claude returns empty `content[]` | Same as API error fallback. |
| Unhandled exception in message processor | "מצטערים, אירעה תקלה זמנית. אנא נסו שנית בעוד רגע 🙏" |
| Webhook 60-second processing timeout | Same error message. |
| Customer sends voice note / image / sticker | "שלום 😊 אני יכולה לעזור רק עם הודעות טקסט. במה אפשר לעזור?" |

### 1.6 Escalation Rules

- `handoff_to_human: true` is set by Claude **only** after **all 4 mandatory fields** are confirmed:
  1. שם מלא (first + last name)
  2. עיר מגורים
  3. טלפון (9–10 digit number)
  4. שעה מועדפת לחזרה
- **Exceptions** that trigger early escalation: emergency (פריצה/חירום), explicit "request for human" by customer.
- Escalation triggers the Google Sheets webhook (if configured). Only fires **once per lead** — `sheets_sent` flag persists to disk.

### 1.7 Test Mode Rules

- When `TEST_MODE=true` and `TEST_PHONE` is set: **only messages from `TEST_PHONE`** are processed. All others are silently blocked.
- If `TEST_MODE=true` but `TEST_PHONE` is empty: all messages are blocked, and a `WARNING` is logged at startup.
- Test commands available from `TEST_PHONE`:
  - `#reset` — clears conversation history + follow-up state for that sender.
- Test leads are written to `leads_test.json` (not `leads.json`).
- Sessions are tracked in `sessions.json` for test mode — see §5.

## 2. Intent and Decision Logic

### 2.1 Two-stage processing

**First message of a conversation:**
The `_detect_scenario()` function runs a regex classifier before Claude is called. If a matching scenario is found, a **pre-written scripted reply** is returned instantly (no Claude API call). The scripted reply has the greeting + company pitch injected into it.

**All subsequent messages:**
Go directly to Claude. The classifier never runs again. Claude has the full conversation history and the system prompt.

### 2.2 Supported Intents (classifier scenarios)

| Scenario key | Trigger condition | Pre-written reply does |
|---|---|---|
| `greeting` | Message matches greeting-only regex | Pitch + "במה אפשר לעזור?" |
| `showroom_address` | כתובת / איפה אתם / איפה האולם / מיקום | Gives address + hours, offers to schedule |
| `showroom_hours` | שעות פעילות / שעות פתיחה / לקבוע פגישה / אולם תצוגה | Gives hours, offers to schedule visit |
| `repair` | תיקון / תקלה / בעיה בדלת / ציר שבור / אחריות | Empathy first + asks what's wrong |
| `mamad` | ממ"ד / חדר ביטחון / מרחב מוגן | Clarifies: new ממ"ד or replacing existing? |
| `frame_removal` | פירוק משקוף (without explicit door type) | Asks entrance or interior first |
| `designed_doors` | Style word (מעוצב/מודרני/קלאסי) without door type | Asks entrance or interior |
| `detailed_inquiry` | "דלת כניסה" + intent word + no style + not a question | Asks: smooth or designed? |
| `detailed_inquiry_interior` | "דלת פנים" + intent word + no style + not a question | Asks: replacement / renovation / new build? |
| `entrance_doors` | "דלת כניסה" present (lower priority) | Guides toward style preference |
| `interior_doors` | "דלת פנים" present (lower priority) | Guides toward project status |
| `vague_inquiry` | Price inquiry without door type | Asks: entrance, interior, or other? |
| `ambiguous` | Generic interest ("מחפש דלת", "מתעניין") | Asks: entrance or interior? |
| *(none)* | Both "דלת כניסה" AND "דלת פנים" in same message | → Goes to Claude for dual-flow handling |
| *(none)* | Everything else | → Goes to Claude |

### 2.3 Keyword / Pattern Matching

All first-message classification uses `re.search()` on the raw lowercase message text.

**Entrance door signals:**
```
דלת כניסה | דלתות כניסה
```

**Interior door signals:**
```
דלת פנים | דלתות פנים
```

**Style signals:**
```
מודרנ | מעוצב | מעוצבת | קלאסי | קלאסית | חלקה | פשוטה
```

**Intent signals:**
```
מתעניין | מתעניינת | מעוניין | מעוניינת | רוצה | צריך | צריכה |
מחפש | מחפשת | מחפשים | מעניין אותי | מעניין אותנו | אשמח | מעוניינים
```

**Frame removal signals:**
```
פירוק משקוף | עם פירוק | בלי פירוק | להוציא משקוף | להחליף משקוף | ללא פירוק
```

**Question signals:**
```
? | יש לכם | האם | אפשר
```

**Greeting-only detection:**
```regex
^(שלום|היי|הי|בוקר טוב|ערב טוב|צהריים טובים|לילה טוב|אהלן|טוב|מה שלומכם|...)$
```

**Closing intent (not in classifier — used mid-conversation):**
```regex
^(תודה רבה|תודה|להתראות|ביי|שלום ולהתראות|יום טוב|קיבלתי את המידע|אצור קשר|...)$
```
→ Only triggers if `conversation_turns >= 2`.

### 2.4 When the bot asks a clarifying question

- **Entrance vs interior unclear:** asks "דלת כניסה, פנים, או משהו אחר?"
- **Style unclear (entrance):** asks "מחפשים משהו חלק ונקי, או עם עיצוב?"
- **Project status unclear (interior):** asks "מדובר בהחלפה, שיפוץ, או בית חדש?"
- **Frame removal:** asks whether frame needs to come out before proceeding
- **Model picked from catalog:** confirms "ההצעה היא על [דגם], נכון?"
- **One question per message** — never stacks.

### 2.5 When it moves directly to contact collection

- Door type + style + project status are all known → moves straight to "מה מספר הטלפון הכי טוב?"
- Contractor / large project → collects name + phone + city quickly, skips style discovery
- Emergency → collects name + phone only, flags handoff immediately
- Explicit "request for human" → collects name + phone only

### 2.6 When it escalates to human follow-up

All 4 mandatory fields confirmed → `handoff_to_human: true` in Claude JSON output → triggers:
1. Google Sheets row append (if `GOOGLE_SHEETS_WEBHOOK_URL` set)
2. Conversation summary generation
3. Lead record closed in `leads.json`

## 3. Conversation Flow Map

### 3.1 דלת כניסה

```
Customer: mentions entrance door
│
├─ First message + pattern match → scripted reply (entrance_doors or detailed_inquiry)
│   ├─ detailed_inquiry: "חלקה או מעוצבת?"
│   └─ entrance_doors: "מה הסגנון — מודרני, קלאסי, מעוצבת?"
│
├─ Style: "חלקה"
│   └─ Bot gently offers designed: "יש לנו מעוצבות עם זכוכית שנראות ממש יפה — מעניין לראות?"
│       ├─ YES → send catalog link + follow-up in ONE message
│       │         https://www.michaeldoors.co.il/catalog/entry-designed
│       └─ NO → proceed to frame removal
│
├─ Style: "מעוצבת" or specific model
│   └─ Send catalog + follow-up in ONE message
│       https://www.michaeldoors.co.il/catalog/entry-designed
│
├─ Frame removal: "יש משקוף קיים שצריך להחליף, או הכל חדש?"
│   └─ Answer noted → proceed to contact collection
│
└─ Contact collection:
    1. Phone
    2. Full name
    3. City
    4. Preferred callback time
    5. Summary readback + "הכל נכון?"
    6. Customer confirms → warm close + handoff_to_human: true
```

### 3.2 דלת פנים

```
Customer: mentions interior door
│
├─ First message → scripted reply (interior_doors or detailed_inquiry_interior)
│   "האם מדובר בהחלפה, שיפוץ, או בית חדש?"
│
├─ Project status answered, style unknown
│   "יש לנו מגוון גדול — חלקה, קו אפס, 4 חריצים, קשת, הזזה ועוד. רוצה שאשלח קטלוג?"
│   ├─ YES → catalog + follow-up in ONE message
│   │         https://www.michaeldoors.co.il/catalog/interior-smooth
│   └─ Style already known → skip catalog
│
├─ Style confirmed → Contact collection (same 4 steps as entrance)
│
└─ NOTE: No frame removal step for interior doors.
   No "מדידה בשטח" promise for interior doors.
```

### 3.3 גם וגם (Both entrance and interior)

```
Customer: mentions BOTH in same message
│
├─ Classifier returns None → Goes to Claude
│
├─ Claude (via system prompt Rule 7):
│   Acknowledges BOTH in opening sentence:
│   "אטפל בשתיהן — לדלת הכניסה... ולדלתות הפנים..."
│   → Handles ENTRANCE DOOR FLOW first
│   → Transitions naturally: "מצוין! עכשיו לגבי דלתות הפנים..."
│   → Handles INTERIOR DOOR FLOW
│
└─ Contact details collected ONCE at end for both
```

### 3.4 דלת מעוצבת

```
Customer: mentions "מעוצבת" without specifying entrance/interior
│
├─ Classifier: _has_style() = true, _has_door_type() = false
│   → designed_doors scenario
│   "יש אצלנו מגוון דלתות מעוצבות... דלת כניסה או דלת פנים?"
│
└─ After answer → joins entrance or interior flow at style-confirmed step
```

### 3.5 פירוק משקוף

```
Customer: mentions frame removal without door type
│
├─ Classifier: _has_frame_removal() = true, _has_door_type() = false
│   → frame_removal scenario
│   "דלת כניסה או דלת פנים, כדי שנוכל לכוון אתכם נכון?"
│
└─ After answer:
    ├─ Entrance → joins entrance flow (frame answer already known, skip that question)
    └─ Interior → frame removal not applicable → joins interior flow
```

### 3.6 הצעת מחיר

```
Customer: "כמה עולה" / "הצעת מחיר" / "מחיר"
│
├─ First message, no door type → vague_inquiry scenario
│   "המחיר תלוי בסוג ועיצוב — דלת כניסה, פנים, או משהו אחר?"
│
├─ First message with door type → joins appropriate entrance/interior flow
│
└─ Mid-conversation, after catalog shown:
    Claude (Rule 7): Checks history for previously expressed model interest
    ├─ Specific model in history → confirms: "ההצעה היא על [דגם], נכון?"
    └─ No model specified → "ראית משהו שמצא חן בעינייך בקטלוג? 😊"
```

### 3.7 אולם תצוגה / כתובת

```
Customer: "איפה אתם" / "כתובת" / "מיקום"
│
└─ showroom_address scenario:
   Address: בעלי המלאכה 15, נתיבות
   Hours: א'–ה' 09:00–18:00 | ו' 09:00–13:00 | שבת סגור
   Offer: "כדאי לתאם מראש — רוצים שאקבע לכם זמן?"
   → If yes: collects name + phone + preferred time for visit booking
```

### 3.8 שעות / תיאום פגישה

```
Customer: "שעות פתיחה" / "לתאם פגישה" / "אולם תצוגה"
│
└─ showroom_hours scenario:
   Hours given → "רוצים לתאם ביקור באולם התצוגה?"
   → If yes: collects name + phone + preferred visit time
```

### 3.9 תיקון / שירות

```
Customer: "תיקון" / "בעיה בדלת" / "ציר שבור" / "אחריות"
│
└─ repair scenario:
   "אוי, אני ממש מצטערת 💙 בואו נטפל בזה מיד. מה בדיוק קורה עם הדלת?"
   → Empathy first, do NOT rush to contact details
   → Let customer explain problem
   → Collect: problem description → city → name + phone → handoff
   → Never promise repair or compensation
```

### 3.10 פנייה כללית / חסרת פירוט

```
Customer: vague interest ("מתעניין", "מחפש דלת", "מה יש לכם")
│
└─ ambiguous scenario:
   "אשמח להבין האם אתם מתעניינים בדלת כניסה או דלת פנים?"
   → After answer: joins appropriate flow
```

### 3.11 Quantity / Multiple Doors / Project Inquiry

```
Customer: mentions quantity ("4 דלתות", "פרויקט", "מוסדי", "כמות")
│
├─ FAQ faq_005 may be injected as context (keyword: "פרויקט", "כמות")
│
└─ Claude handles per Rule 7 (contractor/large project):
   "פרויקט גדול — מנהל המכירות יטפל ישירות 🌟"
   → Collect name + phone + city + scope quickly
   → handoff_to_human: true
```

## 4. Runtime Architecture

### 4.1 Request path (webhook mode)

```
WhatsApp customer
       │
       ▼
Green-API gateway (cloud)
       │  POST /webhook
       ▼
main.py — FastAPI webhook handler
  • Deduplication check (_is_duplicate)
  • Text extraction
  • Non-text detection → send _NON_TEXT_MSG
  • asyncio.wait_for(_process_message(...), timeout=60s)
       │
       ▼
main.py — _process_message(sender, text)
  • TEST_MODE gate (block non-test senders)
  • #reset command handling
  • Follow-up timer reset
  • Closing intent check (is_closing_intent)
  │    └─ YES → get_closing_message() → green.send_message()
  │
  ▼ (not a closing)
simple_router.py — get_reply(sender, text, api_key)
  • Input sanitization (_sanitize_input)
  • History validation (_validate_history)
  • First message? → _detect_scenario() → scripted reply (no Claude)
  • Otherwise → _build_system() (system prompt + FAQ context + business context)
  • Claude API call with JSON prefill
  • _parse_response() → structured dict
       │
       ▼
main.py — _record_lead() → leads.json
main.py — _maybe_send_to_sheets() → google_sheets.py → GOOGLE_SHEETS_WEBHOOK_URL
main.py — green.send_message(sender, reply_text)
       │
       ▼
Green-API gateway → WhatsApp customer
```

### 4.2 Polling loop (fallback, runs in parallel)

When Green-API webhook is not configured, or as a parallel fallback:
```
main.py — _poll_loop() (supervised background task)
  → green.receive_notification()
  → deduplication check
  → same _process_message() path as above
  → green.delete_notification(receipt_id)
```

### 4.3 Follow-up loop (supervised background task)

```
main.py — _followup_loop() (supervised, ticks every 60s)
  → For each sender in _followup where not closed:
      silence >= 15 min → get_followup_message() → green.send_message()
      silence >= 22 min → _CLOSE_MSG → green.send_message() → _attach_summary()
```

### 4.4 File responsibilities

| File | Responsibility |
|------|----------------|
| `src/michael_doors_bot/main.py` | FastAPI app, webhook handler, poll loop, follow-up loop, background task supervision, lead persistence, session management, error tracking, `/diag` endpoint |
| `src/michael_doors_bot/engine/simple_router.py` | Scenario classifier, Claude API calls, conversation history management, JSON response parsing, follow-up/closing/summary generation |
| `src/michael_doors_bot/providers/greenapi.py` | Green-API HTTP client: send message, receive notification, delete notification |
| `src/michael_doors_bot/providers/google_sheets.py` | Single function: POST to Google Sheets webhook URL with lead row data |
| `src/michael_doors_bot/config.py` | Environment variable loading, startup validation, API URL construction from instance ID |
| `src/prompts/systemPrompt.txt` | Complete instruction set for Claude (identity, tone, rules, product catalog, output format) |
| `src/data/faqBank.json` | 56 FAQ entries with keywords, categories, and answers injected as context into Claude when keywords match |

## 5. Session Behavior

### 5.1 Conversation continuity

- Conversation history per sender stored in `_conversations` dict (in-memory), persisted to `conversations.json`.
- History limit: **40 turns** (20 back-and-forth exchanges). Oldest entries trimmed when exceeded.
- History format: `[{"role": "user"|"assistant", "content": "..."}]`
- On each Claude call, the full history is passed + a JSON prefill assistant turn (`"{"`) to force valid JSON output.

### 5.2 Test mode session lifecycle

```
#open  (or any first message from TEST_PHONE in test mode)
   → _open_session(sender) — clears history, records timestamp in sessions.json

Any subsequent message from TEST_PHONE
   → _touch_session(sender) — updates timestamp to prevent 30-min timeout

#reset
   → Clears conversation history + follow-up state + summary_attached set
   → Bot replies: "שיחה אופסה ✓"

#endtest (if implemented in config)
   → _close_session(sender) — removes from sessions.json

Session timeout: 30 minutes of inactivity
   → Session silently expires, next message starts fresh
```

### 5.3 Follow-up state

Each sender has a follow-up state dict:
```json
{
  "last_bot_time": 1714000000.0,
  "followup_sent": false,
  "followup_time": 0.0,
  "closed": false
}
```

- Reset when customer sends a message (`_followup_reset`)
- Updated when bot sends a reply (`_followup_mark_bot_replied`)
- `closed: true` → follow-up loop ignores this sender
- Hourly cleanup: entries closed > 2 hours old are removed from memory

## 6. Lead Data Structure

### 6.1 What the bot extracts from Claude's JSON output

Every Claude response returns a structured JSON object. The following fields are extracted and persisted:

| Field | Type | Description | When set |
|-------|------|-------------|----------|
| `reply_text` | string | Hebrew message to send to customer | Every turn |
| `handoff_to_human` | bool | All 4 fields confirmed, ready for sales team | After 4 fields collected + confirmed |
| `summary` | string | One-line English CRM note (internal only) | Every turn |
| `full_name` | string\|null | Customer's first + last name | When customer provides it |
| `phone` | string\|null | Callback phone (digits only) | When customer provides it |
| `city` | string\|null | Customer's city | When customer provides it |
| `preferred_contact_hours` | string\|null | Exact preferred callback time | When customer states it |
| `needs_frame_removal` | bool\|null | Whether existing frame must be removed | When customer confirms |
| `needs_installation` | bool\|null | Whether installation is needed | When customer confirms |
| `service_type` | string\|null | One of 8 predefined service types | As soon as clear |

**`service_type` values:**
- `"דלת כניסה חלקה"`
- `"דלת כניסה מעוצבת"`
- `"דלתות פנים"`
- `"ממ\"ד"`
- `"דלת מוסדית"`
- `"תיקון/אחריות"`
- `"פרזול וידיות"`
- `"תיאום ביקור אולם"`
- `"אחר"`

### 6.2 Lead record in leads.json

```json
{
  "972521234567@c.us": {
```

## 7. Error Handling and Resilience

### 7.1 Fallback behavior summary

| Failure point | What happens | Customer sees |
|---|---|---|
| Claude API timeout (50s per attempt) | Retry up to 3× with exponential backoff (5s, 10s) | If all 3 fail: error message |
| Claude rate limit | Same retry logic | Same |
| Claude returns invalid JSON | `_parse_response` catches, returns `_PARSE_ERROR_REPLY` | Error message |
| Claude returns `content: []` | `if not response.content: raise ValueError` | Error message |
| Claude returns empty `reply_text` | Detected after parse, substituted with `_PARSE_ERROR_REPLY` | Error message |
| Green-API send fails | `_record_error("send_fail")`, logged `[BOT:SEND_FAIL]` | Nothing sent — customer gets silence (known risk) |
| Google Sheets write fails | Logged, silently skipped. Retry on next conversation update via `sheets_sent=False` check. | No customer impact |
| Unhandled exception in `_process_message` | `[BOT:UNHANDLED]` logged, fallback message sent | Error message |
| Webhook 60s timeout | `asyncio.wait_for` raises `TimeoutError`, fallback sent | Error message |
| Duplicate webhook/poll event | Deduplicated via `_processed_ids_set` — second delivery ignored | No double reply |
| Non-text message (voice, image, sticker) | `_NON_TEXT_MSG` sent | "אני יכולה לעזור רק עם הודעות טקסט" |
| Message > 2000 chars | Truncated at 2000 before Claude call | Reply based on truncated content |
| Poll loop HTTP error | Exponential backoff: 5s → 10s → 20s → ... → 120s max | No customer impact |
| Background task crash | `_supervised()` restarts it after 5s | Brief gap in follow-up monitoring |
| `conversations.json` missing/corrupt | Starts with empty history | Fresh conversation |
| `leads.json` missing/corrupt | Starts fresh | Fresh lead tracking |
| System prompt file missing | `_SYSTEM_PROMPT = ""`, logged as `[BOOT] FATAL` | All Claude replies will be wrong |
| FAQ bank file missing/corrupt | `_faq_bank = []`, FAQ matching disabled | Bot works without FAQ context |
| Missing env vars at startup | `sys.exit(1)` with readable message listing all missing keys | Bot never starts |

### 7.2 What happens on missing JSON fields

`_parse_response()` uses `.get()` with safe defaults for all fields. Missing fields default to `null`/`False`. The `reply_text` is the only hard requirement — if empty, replaced with fallback.

## 8. Known Risks / Open Weaknesses

### 8.1 Critical risks (can cause customer silence)

| Risk | Details | Mitigation in place? |
|------|---------|---------------------|
| **Green-API send failure with no retry** | If `send_message` fails, the bot logs the error but the customer gets silence. There is no retry for sends. | No — just logging. |
| **System prompt file deleted/empty on deploy** | Bot starts, Claude is called, but with empty instructions — all replies will be wrong. Not immediately obvious. | Partial — `[BOOT] FATAL` log, but bot still runs. |
| **Render ephemeral filesystem** | `leads.json`, `conversations.json`, `sessions.json` are written to disk. Render's free tier restarts wipe the disk. | Known limitation — Render Starter tier has persistent disk, but a crash-restart loses in-progress conversation state. |

### 8.2 Behavioral risks (can produce wrong replies)

| Risk | Details |
|------|---------|
| **Claude hallucinates prices** | The system prompt forbids price quotes, but Claude can still produce them, especially when the system prompt is partially matched or history is very long and the instruction is buried. |
| **JSON prefill + long response = truncation** | `max_tokens=900` is a hard ceiling. Very long replies (e.g., full catalog + question + contact request) may be cut mid-JSON, causing parse failure and a fallback reply. |
| **Closing intent false positive** | A customer saying "הכל ברור" or "מצוין תודה" mid-conversation might trigger the closing path. The `is_closing_intent` regex intentionally excludes "הכל ברור" for this reason, but edge cases remain. |
| **Gender switch on long conversations** | Claude is instructed to lock gender, but over 40 turns it may drift — especially if the customer's messages become very short and lack gender markers. |
| **Dual-flow (both door types) relies entirely on Claude** | The classifier returns `None` for dual-flow messages. Claude must handle the sequencing correctly with no scripted scaffolding. Edge cases (customer mentions 5 door types at once) are unpredictable. |
| **Catalog link hardcoded in system prompt** | If the website URL changes, the old links stay in the bot until the system prompt is manually updated and the server is restarted. |

### 8.3 Infrastructure risks

| Risk | Details |
|------|---------|
| **No webhook signature validation** | The `/webhook` endpoint accepts any POST without verifying it's genuinely from Green-API. A malicious actor who knows the URL can send fake messages. |
| **Dedup cache lost on restart** | `_processed_ids_set` is in-memory only. After a Render restart, the last ~500 processed IDs are forgotten. A webhook retry within the restart window could cause double-processing. |
| **Poll loop + webhook both running** | Both paths call `_process_message`. Deduplication prevents double replies, but the poll loop is an unnecessary load if webhook is configured. No automatic detection of "webhook is working, disable poll". |
| **`_sender_locks` grows unbounded** | One `asyncio.Lock` per sender, never cleaned up until hourly cleanup removes closed conversations. Under sustained abuse, this dict could grow large. |
| **No rate limiting per sender** | A single sender can flood the bot with messages. Each message acquires the per-sender lock and queues. Long queue = delayed replies for legitimate customers. |

### 8.4 What should be tested before fully trusting this bot

- [ ] Verify bot does not expose raw JSON to customers under any load condition
- [ ] Verify closing flow: Part A (readback) → customer confirms → Part B (warm close) — both parts must fire
- [ ] Verify follow-up and auto-close timing in a live test conversation with real silence
- [ ] Verify Google Sheets receives exactly one row per lead, even after a Render restart
- [ ] Test with a customer who sends messages in Russian — confirm language switch works
- [ ] Test with a customer who sends > 2,000 characters — confirm truncation and reply
- [ ] Test with a customer who sends an audio message — confirm polite rejection
- [ ] Test a conversation where customer provides all 4 fields in one message (edge case for field extraction)
- [ ] Confirm `/diag` shows `system_prompt_loaded: true` and `faq_count: 56` after each deploy
- [ ] Verify `sheets_sent` persists after a Render restart and prevents duplicate Sheets rows
